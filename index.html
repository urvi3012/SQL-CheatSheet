WHERE vs HAVING — when to filter
⚠ trap▶

WHERE filters rows before aggregation. HAVING filters groups after. Wrong choice = silent wrong results.

WHERE — row-level filter
SELECT dept, COUNT(*)
FROM employees
WHERE salary > 50000
GROUP BY dept
HAVING — group-level filter
SELECT dept, COUNT(*)
FROM employees
GROUP BY dept
HAVING COUNT(*) > 5
Rule: Filtering on an aggregate (SUM, COUNT, AVG)? → HAVING. Everything else → WHERE.
JOIN types — all four with when to use
core▶
INNER JOIN — only matching rows in both
FROM orders o JOIN users u ON o.user_id = u.id
-- drops rows with no match on either side
LEFT JOIN — all left rows, NULLs for unmatched right
FROM users u LEFT JOIN orders o ON u.id = o.user_id
-- anti-join: add WHERE o.id IS NULL → never ordered
CROSS JOIN — every combination (cartesian)
FROM dates d CROSS JOIN products p
-- build a spine → LEFT JOIN actuals to fill gaps
SELF JOIN — table joined to itself
FROM employees e1
JOIN  employees e2 ON e1.manager_id = e2.id
-- e1 = employee row, e2 = their manager
CTEs (WITH) — including recursive
core▶
Standard CTE
WITH active AS (
  SELECT * FROM users
  WHERE active = true
),
orders_cnt AS (
  SELECT user_id, COUNT(*) n
  FROM orders GROUP BY 1
)
SELECT a.*, o.n
FROM active a
LEFT JOIN orders_cnt o
  ON a.id = o.user_id
Recursive CTE — org tree
WITH RECURSIVE org AS (
  -- base: root nodes
  SELECT id, name,
    manager_id, 1 AS lvl
  FROM employees
  WHERE manager_id IS NULL
  UNION ALL
  -- recursive: reports
  SELECT e.id, e.name,
    e.manager_id, o.lvl+1
  FROM employees e
  JOIN org o
    ON e.manager_id = o.id
)
SELECT * FROM org
ORDER BY lvl
When to recurse: org charts, category trees, graph traversal. The base case (WHERE manager IS NULL) anchors the loop.
CASE WHEN — pivot & conditional aggregation
core▶
Input: inspections
type       score
Routine    3
Complaint  7
Routine    8
Complaint  2
Pivot query
SELECT
  SUM(CASE WHEN
    type='Routine'
    THEN 1 ELSE 0
  END) routine_cnt,
  AVG(CASE WHEN
    type='Routine'
    THEN score
  END) routine_avg
FROM inspections
Output + the ELSE trap
routine_cnt  routine_avg
2            5.5
-- ELSE 0   → SUM/COUNT counts all rows
-- ELSE NULL → AVG ignores nulls (omit ELSE for AVG!)
UNION / UNION ALL / INTERSECT / EXCEPT
⚠ trap▶
UNION ALL — combine, keep dupes (fast)
SELECT id FROM table_a
UNION ALL
SELECT id FROM table_b
-- default choice — no extra DISTINCT pass
UNION — combine, deduplicate (slow)
SELECT id FROM table_a
UNION
SELECT id FROM table_b
-- runs DISTINCT internally — avoid if avoidable
INTERSECT — in both tables
SELECT id FROM jan_active
INTERSECT
SELECT id FROM feb_active
-- retained both months
EXCEPT — in A but not B (churn)
SELECT id FROM jan_active
EXCEPT
SELECT id FROM feb_active
-- churned after Jan (MINUS in Oracle)
COALESCE vs NULLIF — safe division pattern
core▶
COALESCE — first non-null value
COALESCE(col_a, col_b, 0)
-- returns col_a if not null, else col_b, else 0
-- use: fill nulls after LEFT JOIN
NULLIF — returns null when two values equal
NULLIF(denominator, 0)
-- returns NULL if denominator = 0 → prevents /0 error
Combined — the safe division pattern (memorise this)
SELECT
  COALESCE(
    SUM(revenue) * 1.0
    / NULLIF(COUNT(*), 0),
  0) AS avg_revenue
-- if COUNT = 0 → NULL → COALESCE gives 0
ROW_NUMBER vs RANK vs DENSE_RANK
⚠ trap▶
All three on tied scores
SELECT name, score,
  ROW_NUMBER() OVER (ORDER BY score DESC) rn,
  RANK()       OVER (ORDER BY score DESC) rk,
  DENSE_RANK() OVER (ORDER BY score DESC) dr
FROM scores
Output — watch the tie on row 2
name   score  rn   rk   dr
Alice  90     1    1    1
Bob    90     2    1    1    ← same score as Alice
Carol  80     3    3    2    ← RANK skips 2, DENSE_RANK doesn't
Dan    70     4    4    3
-- ROW_NUMBER → always unique, arbitrary tiebreak
-- RANK       → gaps after ties (1,1,3,4...)
-- DENSE_RANK → no gaps after ties (1,1,2,3...)
Interview pattern: "latest record per user" → ROW_NUMBER + PARTITION BY. "Top N allowing all ties" → DENSE_RANK.
LAG / LEAD — row offset patterns
advanced▶
Syntax
LAG(col, offset, default)
  OVER (PARTITION BY grp ORDER BY dt)
-- offset: rows back (default 1)
-- default: value when no prior row
Day-over-day diff
SELECT dt, revenue,
  LAG(revenue,1,0)
    OVER(ORDER BY dt) prev,
  revenue - LAG(revenue,1,0)
    OVER(ORDER BY dt) diff
FROM daily_sales
Output
dt        rev  prev  diff
Jan 1     100  0     100
Jan 2     120  100   20
Jan 3     90   120   -30
Session detection with LEAD
CASE WHEN
  LEAD(ts) OVER (
    PARTITION BY user_id
    ORDER BY ts) - ts
  > INTERVAL '30 min'
THEN 'new_session'
ELSE 'same' END
PARTITION BY vs GROUP BY
⚠ trap▶
GROUP BY — rows collapse to 1 per group
SELECT dept, AVG(salary)
FROM employees
GROUP BY dept
-- returns 1 row per dept
PARTITION BY — rows stay, aggregate attached
SELECT name, dept, salary,
  AVG(salary) OVER
    (PARTITION BY dept) dept_avg
FROM employees
-- 1 row per employee + their dept avg
Use PARTITION BY when you need "each row alongside its group's aggregate" — e.g. employee's salary as % of dept total.
FIRST_VALUE / LAST_VALUE — the frame trap
⚠ trap▶
FIRST_VALUE — works fine by default
FIRST_VALUE(order_date) OVER (
  PARTITION BY user_id
  ORDER BY order_date
)
LAST_VALUE — broken by default!
-- Default frame stops at CURRENT ROW
-- so LAST_VALUE = current row, always
-- Fix — always add explicit frame:
LAST_VALUE(col) OVER (
  PARTITION BY user_id
  ORDER BY order_date
  ROWS BETWEEN
    UNBOUNDED PRECEDING
    AND UNBOUNDED FOLLOWING
)
LAST_VALUE is the #1 window function interview trap. Without the full frame clause, it always returns the current row's value — not the last in the partition.
ROWS vs RANGE — the most confused pair
⚠ trap▶
ROWS — physical row count
SUM(val) OVER (
  ORDER BY date
  ROWS BETWEEN 2 PRECEDING
           AND CURRENT ROW
)
-- always exactly 3 rows
-- ignores date gaps
-- use: evenly spaced data
RANGE — logical value range
SUM(val) OVER (
  ORDER BY date
  RANGE BETWEEN
    INTERVAL '2 days' PRECEDING
    AND CURRENT ROW
)
-- all rows within date range
-- handles gaps correctly
-- use: sparse / real time data
Concrete difference — sparse dates
date   val   ROWS(2 prec)   RANGE(2 days)
Jan 1  10    10             10
Jan 2  20    30             30
Jan 5  30    60             30   ← Jan3,4 missing — RANGE sees only Jan5
Jan 6  40    90             70   ← RANGE sees Jan5+Jan6 only
Financial data with weekends/holidays → RANGE. Click-stream or sequential IDs → ROWS.
Rolling SUM / AVG / COUNT — formula to remember
advanced▶
7-day rolling revenue (N=7 → 6 PRECEDING)
SELECT dt, revenue,
  SUM(revenue) OVER (
    ORDER BY dt
    ROWS BETWEEN 6 PRECEDING
             AND CURRENT ROW
  ) rolling_7d,
  AVG(revenue) OVER (
    ORDER BY dt
    ROWS BETWEEN 6 PRECEDING
             AND CURRENT ROW
  ) avg_7d
FROM daily_revenue
Output (rolling window builds up)
dt     rev  rolling_7d
Jan1   100  100
Jan2   200  300
Jan3   150  450
Jan4   180  630
Jan5   120  750
Jan6   200  950
Jan7   90   1040
Jan8   110  1050 ← Jan1 drops off
-- Formula: N-day = (N-1) PRECEDING
Conditional aggregation inside a window
advanced▶

Filter which rows contribute to a rolling aggregate without filtering the result set — CASE returns NULL, SUM ignores NULLs.

Rolling order revenue only (not visits)
SELECT dt, event_type, amount,
  SUM(CASE WHEN
    event_type = 'order'
    THEN amount
  END) OVER (
    ORDER BY dt
    ROWS BETWEEN 2 PRECEDING
             AND CURRENT ROW
  ) rolling_orders
FROM events
Output
dt  type     amt  rolling_orders
1   order    10   10
2   visit    5    10   ← visit = NULL
3   order    20   30
4   visit    8    30   ← still 30
5   order    15   35   ← 20+15
Window directions — past / future / centered
advanced▶
Three frame directions
ROWS BETWEEN 1 PRECEDING
         AND CURRENT ROW
-- backward → trailing history

ROWS BETWEEN CURRENT ROW
         AND 1 FOLLOWING
-- forward → look-ahead

ROWS BETWEEN 1 PRECEDING
         AND 1 FOLLOWING
-- centered → noise smoothing
Input 10,20,30,40 → SUM outputs
val  past    future  centered
10   10      30      30
20   30      50      60
30   50      70      90
40   70      40      70
NTILE vs PERCENTILE_CONT vs PERCENTILE_DISC
⚠ trap▶
NTILE — tag each row with a bucket label
NTILE(4) OVER (ORDER BY spend) AS quartile
-- Q1 = bottom 25%, Q4 = top 25%
-- returns a LABEL per row, not a value
-- use: "find bottom 20% of users"
PERCENTILE_CONT — interpolated threshold value
PERCENTILE_CONT(0.9)
  WITHIN GROUP (ORDER BY spend) AS p90
-- returns a VALUE e.g. 482.5 (may not exist in data)
-- use: "what spend is the p90 cutoff?"
PERCENTILE_DISC — nearest actual data point
PERCENTILE_DISC(0.5)
  WITHIN GROUP (ORDER BY spend) AS median
-- always returns existing row value (no interpolation)
-- CONT → 42.5;  DISC → 42 or 43
Mental model: NTILE → "label each row". PERCENTILE → "give me the threshold value". They answer different questions.
EXISTS vs IN — the NULL trap
⚠ trap▶
IN — evaluates full subquery first
SELECT * FROM orders
WHERE user_id IN (
  SELECT id FROM users
  WHERE country = 'US'
)
-- ⚠ NOT IN with a NULL in list
-- returns ZERO rows — silent bug!
EXISTS — short-circuits, NULL-safe
SELECT * FROM orders o
WHERE EXISTS (
  SELECT 1 FROM users u
  WHERE u.id = o.user_id
    AND u.country = 'US'
)
-- stops at first match
-- handles NULLs correctly
Critical: NOT IN with a NULL anywhere in the subquery returns ZERO rows. Always prefer NOT EXISTS when NULLs might exist — this is a frequent interview bug question.
STRING_AGG / ARRAY_AGG — dialect map
advanced▶
Collect products per order
SELECT order_id,
  STRING_AGG(product, ', '
    ORDER BY product) products,
  ARRAY_AGG(product_id
    ORDER BY product) id_list
FROM order_items
GROUP BY order_id
Output + dialect map
id  products           id_list
101 Eggs, Milk, Yogurt [3,7,12]
-- PostgreSQL: STRING_AGG(col, ',')
-- BigQuery:   STRING_AGG(col, ',')
-- MySQL:      GROUP_CONCAT(col)
-- Snowflake:  LISTAGG(col, ',')
-- Spark SQL:  COLLECT_LIST(col)
QUALIFY — filter window results inline
DE-specific▶

QUALIFY is to window functions what HAVING is to GROUP BY — filters after window computation, no subquery needed.

Without QUALIFY (verbose subquery wrap)
SELECT * FROM (
  SELECT *,
    ROW_NUMBER() OVER (
      PARTITION BY user_id
      ORDER BY order_date DESC
    ) rn
  FROM orders
) WHERE rn = 1
With QUALIFY (clean, one level)
SELECT *
FROM orders
QUALIFY
  ROW_NUMBER() OVER (
    PARTITION BY user_id
    ORDER BY order_date DESC
  ) = 1
-- latest order per user
Available in: Snowflake, BigQuery, DuckDB. Not standard PostgreSQL/MySQL. The cleanest answer for "get latest record per group".
LATERAL JOIN — row-by-row subquery
DE-specific▶

LATERAL lets a subquery reference columns from the outer FROM clause — like a correlated subquery but in the table position. Called CROSS APPLY in SQL Server.

Top 3 orders per user (no window fn needed)
SELECT u.id, u.name, t.*
FROM users u,
LATERAL (
  SELECT * FROM orders o
  WHERE o.user_id = u.id
  ORDER BY amount DESC
  LIMIT 3
) t
-- references u.id inside subquery
Explode array column into rows
SELECT u.id, tag
FROM users u,
LATERAL UNNEST(u.tags) AS tag
-- one row per tag
-- equivalent to EXPLODE() in Spark SQL
Date spine / GENERATE_SERIES — fill gaps
DE-specific▶

Generate a continuous date range, then LEFT JOIN your data onto it. Standard pattern for ensuring every date appears in time-series reports.

PostgreSQL
WITH spine AS (
  SELECT generate_series(
    '2024-01-01'::date,
    '2024-01-31'::date,
    '1 day'::interval
  )::date AS dt
)
SELECT s.dt,
  COALESCE(d.revenue, 0) rev
FROM spine s
LEFT JOIN daily_revenue d
  ON s.dt = d.dt
BigQuery
WITH spine AS (
  SELECT dt FROM UNNEST(
    GENERATE_DATE_ARRAY(
      '2024-01-01',
      '2024-01-31',
      INTERVAL 1 DAY
    )
  ) AS dt
)
SELECT s.dt,
  COALESCE(d.revenue, 0) rev
FROM spine s
LEFT JOIN data d
  ON s.dt = d.dt
Always COALESCE to 0 after the LEFT JOIN — missing days return NULL otherwise and silently break aggregates downstream.
SQL execution order — what runs when
DE-specific▶

Understanding execution order explains why you can't use aliases in WHERE, why window fns must go in SELECT or HAVING, and why QUALIFY exists.

Order of operations (1 = first)
1. FROM / JOIN          -- source tables
2. WHERE                -- row filter (no aggregates yet)
3. GROUP BY             -- group rows
4. HAVING               -- filter groups
5. SELECT               -- compute columns + window fns
6. QUALIFY              -- filter window fn results
7. DISTINCT             -- dedup
8. ORDER BY             -- sort (aliases usable here)
9. LIMIT / OFFSET      -- paginate
Common mistake: Using a SELECT alias in WHERE — fails because WHERE runs before SELECT. You must repeat the expression or wrap in a CTE/subquery.
DATE_TRUNC — time bucketing
core▶
Truncation levels
DATE_TRUNC('week',  ts) → Monday of week
DATE_TRUNC('month', ts) → 1st of month
DATE_TRUNC('year',  ts) → Jan 1 of year
DATE_TRUNC('hour',  ts) → start of hour
Weekly cohort grouping
SELECT
  DATE_TRUNC('week', created_at) cohort,
  COUNT(DISTINCT user_id) new_users
FROM users
GROUP BY 1
ORDER BY 1
DATE_TRUNC vs TO_CHAR: Use DATE_TRUNC for grouping — it preserves sortability and enables date arithmetic. TO_CHAR only for display.
Date arithmetic — dialect differences
⚠ trap▶
PostgreSQL
date1 - date2                      -- integer days
EXTRACT(DAY FROM ts1 - ts2)        -- float days
created_at + INTERVAL '7 days'    -- add time
NOW() - INTERVAL '1 month'
BigQuery / Snowflake
-- BigQuery:
DATE_DIFF(date1, date2, DAY)
DATE_ADD(dt, INTERVAL 7 DAY)

-- Snowflake:
DATEDIFF('day', date1, date2)
DATEADD('day', 7, dt)
Epoch — universal seconds between events
EXTRACT(EPOCH FROM event_b)
  - EXTRACT(EPOCH FROM event_a)
-- seconds since 1970-01-01, works cross-DB
TO_CHAR — date formatting reference
core▶
Tokens (input: 2020-05-01 14:35:20)
YYYY → 2020    YY   → 20
MM   → 05      Mon  → May
MONTH→ MAY     DD   → 01
Day  → Friday  HH24 → 14
HH12 → 02      MI   → 35
SS   → 20
Common format strings
TO_CHAR(ts, 'YYYY-MM-DD')  → 2020-05-01
TO_CHAR(ts, 'Mon YYYY')    → May 2020
TO_CHAR(ts, 'DD Mon YYYY') → 01 May 2020
TO_CHAR(ts, 'HH24:MI:SS')  → 14:35:20
Timezone handling — the UTC midnight bug
DE-specific▶
PostgreSQL
ts AT TIME ZONE 'UTC'
   AT TIME ZONE 'America/Los_Angeles'

-- Or:
TIMEZONE('America/New_York', ts)
BigQuery — group by LOCAL date
DATE(ts, 'America/Los_Angeles') local_dt
-- ⚠ grouping by UTC date in US product
-- midnight cutoff = 4pm–8pm local time
-- extremely common DE pipeline bug!
Rule: Store all timestamps as UTC. Convert to local time only at query/display time. Always specify timezone when grouping by date for user-facing products.
